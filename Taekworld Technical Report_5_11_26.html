Scope

기술 감사(Tech Audit) 는 건물 안전 점검과 같습니다 — 전문가가 잠금 장치, 비상구, 전기 배선, 소화기 위치를 하나씩 확인하고 위험한 곳에 표시를 한 뒤 우선순위를 매기는 작업.

점검한 3개 시스템

시스템비유실제 역할규모감사 기준
API (백엔드)건물 관리실 + 금고결제, 사용자 정보, 모든 비즈니스 규칙약 18만 줄 (.NET)Taekworld/API audit
WEB (웹사이트)정문 + 로비 + 엘리베이터사용자 화면 + 관리자 인트라넷약 12만 줄 (Next.js)Taekworld/WEB audit
App (모바일 앱)사용자의 외출 카드iOS / Android 모바일 앱약 2만 줄 (Flutter)Taekworld/App audit

점검 방법 · 2-track

  • Track A — 외부 사례 조사: 다른 회사들이 비슷한 문제를 어떻게 풀었는지 공개 자료로 비교
  • Track B — 실제 코드 점검: Gitleaks / Semgrep / OSV-Scanner / Trivy / /cso daily 14단계 자동 파이프라인 + Codex(GPT-5) + Gemini 2.5 Pro 두 AI 교차 검증

두 결과를 합쳐서 신뢰도 가중치를 매긴 결론을 냈습니다. 모든 발견에는 "어느 파일 몇 번째 줄" 까지 증거가 보존됩니다.

  • 일반 페이지 + 인트라넷(staff/admin 도구) 두 영역
  • 인트라넷이 VoIP 통화, CRM 활동, 도장주 관리, 비밀번호 재발급 같은 강한 권한 기능 — "정문" 인 줄 알았는데 사실은 마스터 키 보관실

  • 한 개 앱처럼 보이지만 실제로는 두 벌의 코드 — Apple/(iOS) + Android/ 각각 완전 독립
  • 같은 기능을 두 번 만들고 두 번 수정해야 함 → 시간이 갈수록 두 빌드가 다르게 진화

§ 3
문제 — 무엇이 위험했나
Findings

가장 큰 6개 카테고리를 ① 현재 상태 → ② 사고 가능성 → ③ 비유의 3단계로 정리. 🚨 = Critical.

3.1🚨 마스터 키와 금고 비밀번호가 코드에 평문 박혀있던 문제 (API + WEB)CRITICAL
① 현재 상태 (감사 시작 시점)

백엔드 appsettings.json + 웹사이트 코드에 약 80개의 외부 회사 키 + 내부 인증 키가 평문 커밋:

  • Stripe sk_live_* (라이브 결제 키) + webhook signing secret
  • Firebase service account RSA private key (사용자 DB 전체 권한)
  • Google Maps API 키 (서버 + 앱 양쪽)
  • Resend (이메일), Twilio (문자), D-ID (영상) 토큰 다수
  • JWT signing key 32바이트 (사용자 신분증 발급 도장)
  • Postgres 비밀번호 Ta3k@@20-25 (DB 직접 접속)
  • 보안 도구의 "예외 목록(.gitleaks-baseline.json)" 이 502줄 — 매일 보안 검사가 초록불로 잘못 나오도록 설정
② 사고 가능성
  • 결제 시스템 임의 조작 — 임의 청구, 환불 발급, 카드 정보 조회
  • Firebase 키로 모든 사용자 데이터 (학생/학부모/관장 PII) 를 R/W + 임의 푸시 알림
  • JWT 도장 노출 → 누구나 "관리자" 신분증 위조 가능
  • DB 비밀번호로 학생 명단/결제 기록 직접 다운로드
  • 이미 실제 사고 발생: 2026-04-29 Twilio 문자 인시던트 — 노출된 시크릿이 가설이 아니라 비용/리스크로 전환된 사례
③ 비유
사무실 마스터 키와 금고 비밀번호 80개를 메모지에 적어 회의실 화이트보드에 붙여둔 상태. 게다가 매일 아침 보안 담당이 "이상 없음" 보고를 올리도록 보안 알람을 502줄짜리 무시 목록으로 길들여 놓은 상태.
3.2🚨 비상문이 잠겨있지 않았던 문제 — 백엔드 인증 백도어 (API)CRITICAL
① 현재 상태

백엔드 로그인 코드(AuthController.cs:551-560) 안에 "디버그용 백도어" 존재:

if (password == "Btc_2025!") {
    // 비밀번호 검증 단계 통째로 건너뜀
}

admin 이메일 + 이 비밀번호 한 줄만 알면 어떤 사용자로도 로그인 가능. 게다가:

  • JWT 신분증의 발급기관/대상자 검증이 // DEBUG 주석과 함께 꺼져 있었음
  • 백엔드가 Cloud Run 의 --allow-unauthenticated 옵션으로 인터넷에 그대로 노출

세 가지가 곱해져 사실상 인터넷 어디서든 admin 토큰 발급 가능한 상태.

② 사고 가능성
  • 임의 사용자로 로그인 → 학생 정보 열람/수정
  • 관리자 권한으로 결제 기록 조작
  • 모든 인증 보호가 무력화
③ 비유
빌딩 정문에 "이 비밀번호만 누르면 24시간 모든 층 무사통과" 라고 적힌 종이가 안내데스크에 붙어있고, 빌딩 자체는 노숙이 가능한 공원 벤치 옆에 세워진 상태.
3.3🚨 결제 시스템 책임 분리가 안 되어 있던 문제 (API)CRITICAL
① 현재 상태

결제 흐름에서 "누가 결제 완료를 인정할 것인가" 가 명확하지 않음:

  • CreatePaymentIntent / ConfirmPayment 가 클라이언트 메타데이터만 보고 사용자 활성화
  • Stripe webhook (payment_intent.succeeded) 과 메타데이터 충돌 시 어느 쪽을 믿는지 규칙 없음
  • webhook 이 generic "결제 완료된 사용자" fallthrough 로 들어와 잘못된 사용자 활성화 가능
② 사고 가능성
  • 다른 사용자로 결제 처리 — A 가 결제했는데 B 의 신청서가 활성화
  • 같은 결제 의도가 두 번 만들어져 중복 청구
  • 결제 실패 후 재시도가 차단되어 사용자가 영원히 결제 못함
  • 환불 처리가 학생 신청서 상태와 어긋나 회계 분쟁
③ 비유
백화점 카드 결제 후 "이 결제 진짜 통과됐는지" 를 손님이 들고 있는 영수증만 보고 판단하는 셈. 카드사 본사에서 직접 보내주는 확인 신호를 안 보고 영수증만 보는 운영.

3.4🚨 D-ID 영상 토큰 6개가 클라이언트 코드에 박혀있던 문제 (WEB)CRITICAL
① 현재 상태

WEB 코드(mastermessagev2.tsx, dailyVideoGenerator.service.ts) 안에 D-ID API 토큰 6개가 클라이언트 번들에 그대로 포함 — 사이트 방문자 누구나 F12 개발자 도구로 토큰 추출 가능. Square access token / Cloudflare 토큰도 docs/이전개발사_분쟁기록/ 폴더에 평문 보관.

② 사고 가능성
  • Taek 명의로 임의 영상 생성 → 콘텐츠 reputation risk + 비용 도용
  • 분쟁 자료 폴더의 시크릿 → 외주 분쟁 시 추가 노출 위험
③ 비유
가게 정문에 "VIP 카드 6장" 을 친절하게 게시판에 붙여놓은 상태.
3.5🚨 무인증 Resend 발송 endpoint (WEB)CRITICAL
① 현재 상태

/api/intro/admin/send-zoom/route.ts 에 ADMIN_PASSWORD 변수는 정의돼 있지만 실제 비교 코드는 주석 처리. 사실상 무인증으로 누구나 Resend 를 통해 info@taekworld.com 명의로 임의 메일 발송 가능.

② 사고 가능성
  • Taek 도메인 명의 phishing/스팸 → Resend domain reputation 손상
  • 향후 정상 메일도 spam 분류
  • 법적 책임 (CAN-SPAM, GDPR)
③ 비유

🚨 COPPA 대응 — production 런칭 전 필수

왜 이 항목이 별도 섹션으로 분리되어 있나

미국에서는 만 13세 미만 아동의 개인정보를 다루는 모든 사업자가 FTC 의 COPPA(16 CFR Part 312) 규정을 반드시 따라야 합니다. Taek 의 핵심 사용자가 만 13세 미만 학생 → COPPA 미준수 = production 런칭 불가.

본 감사에서 발견된 빨강 20건은 모두 "보안" 영역. 본 §6 은 "법적 컴플라이언스" 영역으로, 기술 사고가 아니라 법적 리스크(FTC 벌금 위반 1건당 최대 USD 51,744 / 2024 기준) 입니다.

6.1 비유

  • 어린이 학원 / 어린이집 / 학교 운영은 건물 안전 점검 외에도 "아이와 관련된 모든 활동을 보호자가 사전 동의해야 한다" 는 별도 법규가 있습니다.
  • 동의 없는 사진 촬영, 무단 외부 영상 송출, 학부모 모르게 외주 업체에 아이 정보 공유 등이 모두 처벌 대상.
  • COPPA 는 이 규정의 온라인 버전. 운영자가 만 13세 미만 아동 정보를 수집·저장·외부 공유하기 전에 검증 가능한 부모 동의(Verifiable Parental Consent, VPC) 를 받아야 한다는 골격.

  3. Plan → Verify → Execute — 5/9 Cloud SQL 인시던트에서 학습 (agent 가 DB 이름 taekworld 로 추정했지만 실제는 postgres → 17분 outage). 이후 모든 production-impact 작업은 추정 0, 사전 확인 필수.
  4. Atomic commit + TDD + Cross-AI gate — git revert <sha> 1줄로 롤백 가능한 변경만 머지. 본 17일간 150+ commit 모두 이 패턴.
  5. 분기별 자동 재감사 — /cso daily 14단계 파이프라인 + gitleaks + osv-scanner + npm audit baseline → trend 추적 → 회귀 자동 알림. 본 통합 리포트는 분기마다 1회 갱신 권장.
